# Story 類型：User Story vs. Enabler Story

## User Story

描述終端使用者可直接感知的價值。

### 格式

```
身為 [受益者角色]，
我需要 [能力/功能]，
以便 [期望成果]。
```

### 範例

**好：**
> 身為老師，我需要透過專屬網址存取 AI 教學工具，以便在課堂中直接使用而不需要額外的帳號設定。

**壞（偽裝成 Story 的任務）：**
> 身為開發者，我需要設定 GCS bucket，以便靜態檔案有地方放。

壞的版本沒有使用者價值，它是一個實作任務。

### 判斷標準

- 受益者是否會因為這個 Story 完成而「感覺到」什麼改變？
- 如果受益者是「開發者」或「系統」，考慮是否應改為 Enabler Story。

---

## Enabler Story

描述技術性工作，不強制使用「身為...」格式，但必須說明它賦能了什麼。

### 格式

```
標題：[簡潔描述這項工作]
價值陳述：[這項工作使得 ___ 成為可能]
驗收標準：[outcome-focused 的完成條件]
```

### 適用場景

| 場景 | 範例 |
|------|------|
| 基礎設施建立 | 部署環境、CI/CD pipeline |
| 安全性強化 | API Key 隱藏、權限控管 |
| 效能改善 | 快取機制、CDN 設定 |
| 技術債清理 | 重構、升級相依套件 |

### 範例

**好：**
> **標題：** API Key 安全化
> **價值陳述：** 使所有 AI 教學工具能安全對外開放，不因 Key 外洩導致費用失控。
> **驗收標準：**
> - 前端原始碼與瀏覽器網路請求中不可見任何 API Key
> - 所有 AI 功能在安全化後仍正常運作

**壞（過度規定實作）：**
> **標題：** 建立 Cloud Run API Proxy
> **驗收標準：** 使用 Cloud Run 部署 Node.js server，從 GCP Secret Manager 讀取 Key...

壞的版本規定了具體技術方案，限制了工程團隊的選擇空間。

---

## 如何選擇？

```
這項工作完成後，終端使用者能直接感受到嗎？
  ├── 是 → User Story
  └── 否 → 這項工作賦能了哪些 User Story？
           ├── 能明確指出 → Enabler Story（附上關聯）
           └── 指不出來 → 重新思考這項工作是否必要
```
