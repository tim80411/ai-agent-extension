# Story Mapping 方法論

> 來源：Jeff Patton, 2005 首次提出, 2014《User Story Mapping》一書系統化

## 核心概念

### 動機：解決 Flat Backlog 的問題

傳統的 Flat Backlog（一維的優先序列表）讓團隊失去全局脈絡——你知道下一個要做什麼，但不知道整個產品的使用者旅程長什麼樣。Story Mapping 透過二維地圖還原全局視野：水平軸是使用者旅程的時間/邏輯順序，垂直軸是每個步驟的實作優先序。

### 三層結構

| 層級 | 名稱 | 定義 | 數量參考 |
|------|------|------|----------|
| 第一層 | **Activity**（活動） | 使用者的高層級目標群組，用動詞短語描述。例如：「管理課程內容」、「完成結帳流程」 | 一個產品通常 5–15 個 |
| 第二層 | **Task**（用戶任務） | Activity 下使用者的具體步驟，按時間或邏輯順序排列 | 一個 Activity 下通常 3–8 個 |
| 第三層 | **Story**（用戶故事） | 每個 Task 的具體實作方式，最小可開發單位 | 每個 Task 下數量不等 |

### 關鍵術語

- **Backbone**（骨幹）：最頂端一行 Activities 構成的水平骨架，代表產品的完整使用者旅程
- **Walking Skeleton**（行走骨架）：每欄最高優先的 Stories 連成的最小端到端可運行版本，即第一條水平切線以上的部分

## 操作步驟（六步法）

### Step 1：框架問題

在動手拆解前，先回答三個問題：

1. **目標使用者**是誰？（具體角色，不是「所有人」）
2. 要**解決什麼問題**？（使用者現在的痛點）
3. **成功是什麼樣子**？（Release 後使用者能做到什麼）

### Step 2：建立 Backbone

識別所有 Activities，由左到右按時間或邏輯順序排列。

原則：**先一英里寬、一寸深**——先覆蓋完整的使用者旅程，不要在任何一個 Activity 上鑽太深。

操作方式：
- 從使用者的角度出發，想像他完成目標的完整流程
- 每個 Activity 用動詞短語描述（例如：「瀏覽商品」而非「商品頁面」）
- 通常 5–15 個 Activities 足以覆蓋一個產品或大功能

### Step 3：探索 Tasks

在每個 Activity 下拆出具體的 Tasks：

- 按時間順序列出使用者會做的事
- 考慮不同角色可能有不同的 Tasks
- 考慮不同路徑（首次使用 vs. 老手、成功路徑 vs. 錯誤處理）

### Step 4：垂直排序

同一欄內的 Stories 按使用者價值 / 業務重要性由上而下排列：

- 最上方 = 最核心、最高優先
- 最下方 = nice-to-have、可延後

### Step 5：定義 Release Outcome 並切片

以 Outcome 為錨點畫水平切線：

- 每條切線代表一個 Release
- 切線以上的 Stories 構成該 Release 的範圍
- **第一條切線 = MVP**（最小可行產品）

### Step 6：精煉第一個 Release

確認第一個 Release 的 Stories：

- 每個 Activity 至少有一個 Story 被包含（確保端到端可運行）
- 每個 Story 可被獨立開發和驗收
- 整體構成有意義的使用者價值

## 拆解技巧

### 如何識別 Activities

- **從使用者目的出發**：使用者來到這個產品要完成什麼？
- **角色扮演**：假裝自己是使用者，走一遍完整流程
- **從訪談提取動詞**：使用者描述中的動詞通常對應 Activities

### 從 Activity 拆 Tasks

| 拆分維度 | 說明 | 範例 |
|----------|------|------|
| 時間順序 | 使用者依序做的事 | 搜尋 → 篩選 → 比較 → 加入購物車 |
| 角色差異 | 不同角色有不同步驟 | 買家：下單；賣家：出貨 |
| 頻率差異 | 一次性 vs. 重複性操作 | 初始設定 vs. 日常使用 |

### 從 Task 拆 Stories

| 拆分維度 | 說明 | 範例 |
|----------|------|------|
| 功能路徑 | Happy path vs. 替代路徑 | 信用卡付款 / 超商付款 / 轉帳付款 |
| 資料複雜度 | 簡單 → 複雜 | 搜尋：關鍵字搜尋 / 進階篩選 / 全文搜尋 |
| 使用者類型 | 不同使用者有不同需求 | 一般使用者檢視 / 管理者檢視 |
| 錯誤情境 | 非正常流程 | 付款失敗重試 / 庫存不足提示 |
| 效能體驗 | 基本 → 優化 | 清單載入 / 無限捲動 / 離線快取 |

## Release Slicing

### 以 Outcome 為錨的切片方法

每個 Release 切片前，回答四個關鍵問題：

1. **目標使用者是誰？**（這個 Release 服務的主要角色）
2. **他們現在怎麼做？**（沒有這個產品/功能前的替代方案）
3. **Release 後他們怎麼做？**（新的使用者行為）
4. **如何衡量成功？**（可觀察、可量測的指標）

### 切片原則

- 第一條切線 = MVP：能跑通完整旅程的最小集合
- 每個 Release 都是端到端可用的（不是只完成某個模組）
- 以「使用者能做到什麼新事」定義 Release，不是以「開發了什麼功能」定義

## 常見錯誤（Jeff Patton 官方 5 個錯誤）

### 1. 失去故事

**症狀**：卡片變成名詞（頁面名稱、模組名稱）而非動詞（使用者行為）。
**修正**：每張卡片都應回答「使用者在做什麼？」用動詞短語描述。

### 2. 迷失在細節

**症狀**：過早深入某個 Activity，其他 Activity 還沒展開。
**修正**：嚴守「一英里寬、一寸深」——先覆蓋全局骨幹，再逐區深入。

### 3. 過度關注流程

**症狀**：把 Story Map 變成流程圖，加入判斷節點和分支箭頭。
**修正**：Story Map 是探索工具，不是流程文件。保持簡單的表格/卡片形式。

### 4. 畫太大

**症狀**：試圖把整個產品的所有功能都放進一張 Map。
**修正**：只畫涉及的使用者旅程。一次聚焦一個主要目標或功能領域。

### 5. Release 沒錨定 Outcome

**症狀**：以開發容量填塞 Release（「這個 Sprint 還能塞幾個 Story」）。
**修正**：先定義 Outcome（使用者 Release 後能做到什麼），再選擇需要的 Stories。

## 何時適合 / 不適合

### 適合使用 Story Mapping

- 新產品或大功能的初期規劃
- 需要建立團隊共識、對齊全局視野
- 需要定義 MVP 或 Release 計劃
- 需求模糊，只有大方向目標
- 多角色場景，需要理清不同使用者旅程

### 不適合使用 Story Mapping

- 小功能修改或 bug fix
- 純技術債重構（無使用者旅程脈絡）
- 需求已經很明確，已有清楚的 Stories 清單
- 單一明確功能的開發

## 範例：線上教學平台 Story Map

以一個線上教學平台為例，展示完整的 Story Map 結構：

```
Activity:  | 探索課程      | 註冊與登入    | 學習課程        | 追蹤進度       | 管理帳戶      |
           |--------------|--------------|----------------|---------------|--------------|
Task:      | 瀏覽課程清單   | 建立帳號      | 觀看影片        | 查看學習紀錄    | 修改個人資料   |
           | 搜尋課程      | 登入         | 閱讀教材        | 查看完成率      | 管理付款方式   |
           | 查看課程介紹   | 忘記密碼重設   | 完成測驗        | 取得完課證書    |              |
           |              |              | 參與討論        |               |              |
           |--------------|--------------|----------------|---------------|--------------|
Stories:   |              |              |                |               |              |
(Release 1)| 瀏覽清單(分類) | Email 註冊   | 播放影片(基本)   | 標記已完成      | 修改姓名/頭像  |
  MVP      | 關鍵字搜尋    | Email/密碼登入| 下一課/上一課    | 查看進度百分比   |              |
-----------|--------------|--------------|----------------|---------------|--------------|
(Release 2)| 進階篩選      | 社群帳號登入   | 調整播放速度     | 完課證書(PDF)  | 變更密碼      |
           | 課程評分顯示   |              | 課程筆記        | 學習時數統計    | 管理信用卡    |
-----------|--------------|--------------|----------------|---------------|--------------|
(Release 3)| 個人化推薦    | 雙因素驗證     | 離線下載觀看     | 學習路徑建議    | 帳戶刪除      |
           | 收藏清單      |              | 即時問答互動     | 社群排行榜      |              |
```

### Release Outcome 定義

- **Release 1 — Outcome**: 使用者能找到課程、註冊帳號、觀看影片並追蹤自己的學習進度
- **Release 2 — Outcome**: 使用者有更豐富的學習體驗（筆記、調速）並能取得完課證明
- **Release 3 — Outcome**: 使用者獲得個人化體驗和社群互動能力
