# 優先序排定方法

## 排序維度

| 維度 | 問題 | 高優先 | 低優先 |
|------|------|--------|--------|
| 依賴性 | 其他 Story 是否依賴它？ | 被多個 Story 依賴 | 無下游依賴 |
| 使用者價值 | 使用者能多快感受到價值？ | 直接可感知 | 間接支撐 |
| 風險 | 不確定性有多高？ | 高不確定性（先做以降低風險） | 已充分理解 |
| 成本 | 實作成本與價值的比值？ | 低成本高價值 | 高成本低價值 |

---

## 排序流程

```
1. 畫出依賴關係圖
   → 找出沒有前置依賴的 Story（這些可以先開始）

2. 在同層級中，按「使用者價值」排序
   → 價值高的優先

3. 價值相近時，按「風險」排序
   → 風險高的優先（早點驗證、早點發現問題）

4. 標記為 P0 / P1 / P2 / P3
   → P0: 必須在本 Sprint 完成，其他 Story 依賴它
   → P1: 本 Sprint 的核心交付
   → P2: 本 Sprint 盡量完成
   → P3: Optional，有時間再做
```

---

## Enabler Story 的特殊處理

Enabler Story 本身沒有直接使用者價值，排序時看它賦能了多少 User Story：

```
賦能 3+ 個 User Story → 與最高優先的被賦能 Story 同級或更高
賦能 1-2 個 User Story → 與被賦能 Story 同級
賦能 0 個 → 質疑是否需要做
```

---

## 注意事項

- 優先序不是固定的，隨著 Sprint 進行可能調整
- 避免所有 Story 都標 P0（如果都是最高優先，等於沒有優先序）
- P0 通常不超過總 Story 數的 30%
