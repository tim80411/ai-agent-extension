# 範圍界定方法

## 核心原則

範圍不是「我們要做什麼」的清單，而是「我們願意在這個時間框架內解決什麼問題」的承諾。

---

## Appetite（胃口）vs. Estimate（估算）

來自 Shape Up 方法論的核心概念。

| | Appetite | Estimate |
|---|---------|----------|
| 問的問題 | 我們願意花多少時間？ | 這要花多少時間？ |
| 誰決定 | PM / 業務 | 工程團隊 |
| 性質 | 預算（固定的） | 預測（變動的） |
| 超出時會 | 砍 scope | 延期 |

**實際操作：** 先決定 appetite，再讓工程團隊在這個框架內決定能做到什麼程度。

---

## No-gos（明確排除項）

每個 Sprint / Story 都應列出「不做什麼」。

### 撰寫原則

- 列出利害關係人可能預期但本次不做的事
- 列出看似相關但會大幅增加複雜度的延伸需求
- 不需要列出顯然不相關的事

### 範例

```
No-gos:
- 本次不處理使用者身份驗證（公開網站，不需登入）
- 本次不建立自動擴展機制（流量尚小）
- 本次不支援自訂 domain（統一使用 *.nani.com.tw）
```

---

## Rabbit Holes（已知風險）

主動標記可能吞噬時間的陷阱，提供應對策略。

### 撰寫格式

```
風險：[描述可能發生的問題]
影響：[如果發生，會造成什麼]
應對：[建議的處理方式]
```

### 範例

```
風險：部分專案的 Gemini API 呼叫使用了 streaming 模式，proxy 轉發 streaming 可能較複雜。
影響：可能導致 2 個專案的遷移時間超出預期。
應對：先完成非 streaming 專案，streaming 專案單獨評估。
```

---

## 範圍界定檢查清單

- [ ] 是否定義了 appetite（願意投入的時間上限）？
- [ ] 是否列出了 no-gos？
- [ ] 是否標記了已知的 rabbit holes？
- [ ] 範圍是否能在一個 sprint 內完成？若否，是否應拆成多個 sprint？
