# 共用脈絡模板

所有 reviewer agent 共享此模板。在 Phase 1 中填入實際值後，注入每個 agent 的 prompt。

## 模板

```markdown
# 共用脈絡

## 專案資訊
- 專案：{專案名稱}
- 技術堆疊：{框架、ORM、資料庫、快取、認證等}
- 程式碼統計：約 {X} 行 TypeScript
- 模組數量：{X} 個功能模組
- 最複雜的檔案：{檔案名稱}（{X} 行、{X} 個依賴）

## 審查範圍
以下為審查目標檔案（依重要性排序）：
{檔案清單，每行一個，格式：- src/path/to/file.ts（簡短說明）}

## 嚴重度評級系統
| 嚴重度 | 定義 | 行動 |
|--------|------|------|
| 嚴重 | 臭蟲/安全性/資料遺失，無法測試/維護 | 阻擋合併 |
| 重要 | 顯著損害品質，數週內會造成實際痛苦 | 強烈建議修正 |
| 輕微 | 最佳實踐違規，即時影響低 | 修正或建立工單 |
| 建議 | 值得考慮的改善 | 作者自行決定 |

## 工作量評級
- 高：需要架構變更或跨多個模組修改
- 中：需要修改多個檔案但範圍明確
- 低：單一檔案或幾行即可修正

## 發現報告模板

每個發現**必須**嚴格遵循此格式：

```
ID:        [視角代碼]-[兩位數編號]
檢查項:    對應的檢查項編號（例如 RD-4）
檔案:      從 repo 根目錄算起的確切路徑
行號:      確切行號（多處用逗號分隔）
標題:      一句話描述問題
嚴重度:    嚴重 | 重要 | 輕微
工作量:    高 | 中 | 低

目前程式碼:
[相關程式碼片段]

建議修正:
[改善後的程式碼或做法]

原因說明:
[為什麼修正更好，對應檢查項目]
```

## 輸出格式要求

每個 reviewer 的輸出必須包含：

1. **摘要表格**：

| ID | 檢查項 | 檔案 | 行號 | 嚴重度 | 工作量 | 摘要 |
|----|--------|------|------|--------|--------|------|
| XX-01 | XX-N | file.ts | 123 | 嚴重/重要/輕微 | 高/中/低 | 一句話 |

2. **詳細發現**：每個發現使用上述模板格式

3. **語言**：繁體中文
```

## 組裝共用脈絡的步驟

Phase 1 中按以下順序收集並填入模板：

1. **專案資訊**：讀取 `package.json` 取得名稱和主要依賴
2. **程式碼統計**：使用 `find src -name '*.ts' -not -path '*/node_modules/*' | xargs wc -l` 統計行數
3. **模組結構**：使用 `ls src/modules/` 列出功能模組
4. **複雜度識別**：找出行數最多的 service 檔案
5. **審查範圍**：根據 Phase 0 確定的範圍列出檔案清單
