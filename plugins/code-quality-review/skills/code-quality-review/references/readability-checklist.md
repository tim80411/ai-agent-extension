# 可讀性 (RD) 檢查項目

## 檢查項清單

| 編號 | 檢查項 | 說明 |
|------|--------|------|
| RD-1 | 命名一致性 | 變數、函式、類別命名是否遵循一致的慣例？是否有拼字錯誤或用詞不統一？同一概念是否使用相同名稱？ |
| RD-2 | 函式長度與複雜度 | 單一函式是否超過合理長度（建議 < 50 行）？是否有過長的方法需要拆分？超過 100 行的函式應標記為「重要」。 |
| RD-3 | 死碼與誤導性程式碼 | 是否有空殼函式（no-op）、永遠回傳固定值的方法、或命名與實際行為不符的程式碼？ |
| RD-4 | 魔術數字與魔術字串 | 是否有未命名的常數值？數字或字串字面值是否應該被提取為具名常數或列舉？特別注意狀態碼、錯誤碼、身份旗標。 |
| RD-5 | 註解與死碼 | 被註解掉的正式程式碼是否有 issue 參考？缺少 issue 參考的註解碼應建議移除。 |
| RD-6 | 例外類型使用 | 是否拋出泛用的 `Error` 或 `new Error()` 而非型別化的領域例外？檢查專案是否有自訂例外體系。 |
| RD-7 | 匯入路徑風格 | 專案內是否混用絕對路徑（`src/`、`@/`）和相對路徑（`../../`）？是否有一致的匯入風格？ |
| RD-8 | 錯誤碼語意 | 錯誤碼值是否具有正確的語意？例如成功碼（0）是否被誤用於錯誤情境？預設錯誤碼是否傳達正確意圖？ |

## 指導原則

### 審查心態
專注於「其他開發者第一次閱讀此程式碼時的理解難度」。以一位新加入團隊的中階開發者作為目標讀者。

### 嚴重度判定指引
- **嚴重**：此視角通常不會產生嚴重發現，除非命名錯誤直接導致臭蟲（例如方法名稱與行為完全相反）
- **重要**：拼字錯誤出現在公開 API（DTO 屬性、方法名稱）、超長函式、語意混淆的錯誤碼
- **輕微**：內部命名不一致、次要的格式問題、私有方法的風格問題

### 合併策略
若同一問題出現在多處，合併為一個發現並列出所有行號。例如：
- 同一個拼字錯誤出現 3 次 → 一個發現，行號列出所有位置
- 魔術數字 `1` 在多處代表「啟用」 → 一個發現

### 特別關注
- 公開 API 中的拼字錯誤（DTO 屬性名稱一旦發布就難以修改）
- 手動欄位對應（30+ 個欄位的手動 mapping 是維護負擔）
- 同一概念有多個不同命名（例如 `validated`、`valided`、`verified`）
- `errorCode: 0` 在錯誤回應中的使用（語意矛盾）
